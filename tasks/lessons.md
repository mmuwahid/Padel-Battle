# Lessons Learned

> Full pre-S041 chronological log preserved at [`_archive/lessons-pre-s041.md`](_archive/lessons-pre-s041.md). This file holds the distilled rule set + most recent durable patterns. Re-distill when this file crosses 30 KB.

## Critical Rules (Always Enforce)

21. **Notification mockups MUST be backed by an explicit DB-side fan-out (trigger or RPC).** Don't ship a notification UI variant without writing the SQL that produces those rows. NotificationCenter rendering ≠ notifications actually being created. (S044 hotfix: admin pending-match notifications had UI but no producer for ~30 minutes post-deploy.)
22. **When adding a column reference for a new feature, grep ALL `select()` queries on that table.** Not just the one you're editing. Multiple selects on the same table are common (single-row vs multi-row); missing one fails silently. Pattern: `grep "from(\"<table>\").select"`. Easier baseline: include `id` in every `select` by default — bytes are cheap, undefined lookups aren't. (S044 hotfix: Promote button invisible because second `league_members` select didn't include `id`.)
23. **Co-locate admin actions on user-generated content with the content surface, not a dedicated dashboard.** Per-match approvals belong on the Matches tab; sidebar dashboards work for league-wide settings (rename, invite codes, roster management). Users discover actions where the content lives. (S044 hotfix: queue moved AdminDashboard → Matches tab inline.)
24. **`bool_and()` over a SELECT IGNORES NULLs — never use it where missing values must count as failure.** Use `count(*) WHERE condition` against the expected total instead. (S044 hotfix: `respond_to_challenge` was returning TRUE after only some players accepted because NULL responses weren't counted.) Pattern: `(SELECT count(*) FROM unnest(arr) AS x WHERE COALESCE(map->>x::TEXT, '') = 'accepted') = array_length(arr, 1)`.
25. **When tightening RLS during a hardening pass, audit every consumer of the affected table.** S025 reduced `league_members_select` to `user_id = auth.uid()` — single-user visibility. Any feature requiring co-member enumeration (Settings → Admin Management list, AdminDashboard Promote/Demote, future leaderboard avatars) silently breaks. Hardening that's too restrictive is also a bug. Pattern: `league_id IN (SELECT get_user_league_ids(auth.uid()))` lets members see members of THEIR leagues — the right scope for a league-scoped app. (S044: 30+ minutes wasted because Promote button rendered fine in dev with seed data but never showed in prod.)
26. **When a feature relies on enumerating rows of a table behind RLS, write a smoke query first.** Before debugging React render gates, run `SELECT count(*) FROM <table> WHERE <league_filter>` as the affected user. If it returns less than expected, fix RLS first. Saves diagnostic-block round-trips. (S044 lesson — should have run this before adding the diagnostic block.)
27. **In-app + push notification is the standard pair for any user-affecting state change.** Whenever an RPC inserts a `notifications` row for another user, also call `sendPushNotification(...)` from the client (or invoke the Edge Function server-side). Audit any future RPC that does `INSERT INTO notifications` against this rule. (S044: `set_member_role` only inserted the in-app row — promoted users never got a device push until polish round added the client-side call.)
28. **For "feels too tall" UX complaints, scale the content first, padding second.** Padding pixel diffs of 4–8px are rarely visually perceptible; reducing fontSize and image dimensions IS. Apple HIG tap targets (32×32 minimum for buttons) are the floor — content typography (logos, headings, subtitles) can scale below this. (S044 took 3 commits to make the header feel smaller because v1 + v2 only touched padding; v3 finally shrank logo 32→26, h1 16→14, subtitle 10→9 — only then did the user perceive a change.)
29. **When applying multi-axis rules to existing data, enumerate ALL axes explicitly per row before recommending actions.** Make a 2D table: rows = data items, columns = each rule independently. Recommend per cell, not per item. Catches "this item passes rule X but fails rule Y" cases that single-axis thinking misses. (S045: initial recommendation flagged matches with FIP-invalid 3rd sets but missed dead-rubber 3rd sets that were FIP-valid shapes — two different rules — until user caught it.)
30. **Postgres CHECK constraints can't contain subqueries — wrap composite shape rules in an IMMUTABLE SQL function and call it from CHECK.** Pattern: `CREATE FUNCTION shape_valid(jsonb) RETURNS boolean LANGUAGE sql IMMUTABLE SET search_path = public AS $$ SELECT ... $$;` then `CHECK (shape_valid(col))`. Function must not depend on table data. (S045: `is_valid_set` + `all_sets_valid` wrapper allowed CHECK to validate every JSONB array element via subquery indirectly.)
31. **Conditional CHECK with status-based bypass for historical data.** Pattern: `CHECK (status = 'archived' OR shape_valid(col))`. Lets you tighten new-row validation without rejecting existing rows that violate the new rule — flip them to the bypass status during a rectification migration first. Trigger-level `RAISE EXCEPTION` on the strict path catches new client-driven violators. (S045: `matches_sets_valid_or_incomplete CHECK (status = 'incomplete' OR all_sets_valid(sets))` lets matches #2 and #5 keep their malformed sets as `status='incomplete'` while enforcing strict shape rules on new approved/pending rows.)
32. **When adding a new step to an existing async flow, grep for `result|response|data` in the function body before naming a new local.** Generic names should be reserved for the LAST mention in a flow. (S045: validator output named `result` clashed with `play_challenge` RPC response `result` in `ScheduleView.saveLoggedMatch`. Renamed to `v` to avoid the redeclaration.)

33. **When restyling a value that another session previously tuned by hand, GREP THE COMMIT MESSAGE FIRST.** I introduced a header padding regression by changing `padding: 4px 16px` to `padding: 6px 16px 10px` and `safe-area+0px` to `safe-area+6px` — silently undoing S044 v3 work where 3 separate commits had shrunk the header to its tightest visually-acceptable values. Pattern: `cd /tmp/Padel-Battle && git log --oneline -- <file> | head -10` AND `git log --all --grep="<search-term>"` before tuning visual values. (S046 v1 → v2 wasted iPhone smoke-test cycle on a regression that was avoidable.)

34. **For a "floating" bottom nav with side gutters, add a SOLID PEDESTAL at z-99 with `pointerEvents: none` to hide content bleed-through.** A floating pill nav with gutters lets scrolled content show through the gaps below and beside it. Solution: a fixed-position div at `bottom: 0; left: 0; right: 0; height: 82px+safe-area; background: BG; zIndex: 99; pointerEvents: none` directly before the nav. The nav stays at zIndex 100 with its rounded outline; pedestal hides everything beneath. `pointerEvents: none` ensures touches pass through to content below, never interfering. (S046 v2 fix — visually identical to non-floating nav from below the nav up, but preserves the floating aesthetic everywhere else.)

35. **When extracting an inline section into a dedicated screen, also strip the now-orphaned state and helper functions from the parent component.** I rewrote AdminDashboard from 270 lines down to ~120 lines because moving Player Management out left behind 5 useState hooks, 2 useMemo hooks, and 3 functions that were only consumed by the moved section. Half-completed extractions are worse than none — left orphan code still has imports, gets bundled, and confuses future readers. Pattern: after moving JSX, grep the parent for any `setX` / `useX` / function definition that's now only referenced inside the comment/dead code. (S046: dropped `adminEditId/Name`, `confirmDeactivate`, `adminLoading`, `confirmRole`, `roleBusy`, `memberIdByUserId`, `roleByUserId`, `updatePlayerName`, `deactivatePlayer`, `setRole` from AdminDashboard — moved to PlayerManagement.)

36. **`avatars/{userId}/...` storage policy doesn't fit "admin manages other users' photos".** The default Supabase avatars bucket policy `(storage.foldername(name))[1] = auth.uid()::text` ties path ownership to the uploader's UID. To allow league admins to upload player photos, add a separate policy keyed on a different first-folder prefix (e.g. `avatars/players/*`). Tradeoff: any authenticated user could upload there too — UI gating + monitoring is the second line. For tighter control, JOIN with `league_members` in the policy USING clause to restrict to admins. (S046: shipped the looser policy for speed; documented as future-tightening candidate.)

37. **For "edit a row in admin dashboard" UX, prefer a bottom-sheet modal over an inline accordion.** Inline edit accordions cause jumpy layout shifts when expanded mid-list and don't scale to multiple form fields (photo + dropdown + toggle). Bottom-sheet modal: full-width on mobile, max-width-cap on desktop, `position: fixed` with `borderTopLeftRadius/RightRadius`, dark overlay tap-to-close, drag-handle visual cue. Native iOS feel. The same pattern used in EditMatchModal (S044) generalized cleanly to EditPlayerModal. (S046: 3-field-or-more edit forms = bottom sheet, sub-3-field edits can stay inline.)

38. **DB-default `playing_position` to NULL, not `'right'`, even though most players are right-handed.** Setting an opinionated default would mark every existing row as right-handed without confirmation. NULL = "not set yet" semantically, matches the user-supplied data state. CHECK constraint allows NULL alongside the enum: `CHECK (playing_position IS NULL OR playing_position IN ('left', 'right'))`. (S046: same principle for `country` — NULL until backfilled, no presumed default.)

39. **An IF NOT EXISTS ALTER TABLE migration that is re-run is fine; an UPDATE migration that is re-run is fine ONLY IF the WHERE clause is row-pinned by ID.** Backfill UPDATEs by player ID (not by name) are idempotent and traceable — re-running them does nothing because the values don't change. UPDATE by name would be unsafe if a name was edited between migration runs. (S046 country backfill: 13 rows, all UPDATE-by-id, idempotent. Pattern matches S045 rectification migration.)

40. **Body/html background paint is the cheap fix when "sticky header looks fine when scrolled but too low at the top".** iOS Safari (PWA standalone) rubber-band overscroll exposes the wrapper element's `background` color above a `position: sticky` header. If the wrapper background differs from the header's gradient start, the perceived header height changes between scrolled-down and absolute-top states. Fix: paint `html, body { background: <gradient-start-color>; overscroll-behavior-y: none; }` — overscroll reveals the same color as the header so the visual is identical at both extremes, AND `overscroll-behavior-y` prevents rubber-band entirely on Chromium / iOS-PWA. One-line CSS, no per-screen tweaks, no restructuring sticky→fixed. (S049 Issue #15 fix — deploy d3200d6.)

41. **Section co-location follows the user's mental model, not the codebase's history.** Admin Management lived in Settings only because that's where the role-dropdown UI was first built (S044 FT-09 sprint). Once the user pointed out it conceptually belongs with Admin Dashboard, the move was a 30-line reshuffle. Pattern: revisit IA periodically — initial component placement is often the path of least resistance during rapid iteration, not the optimum for discovery. Same principle as the S044 hotfix that moved Approvals queue from AdminDashboard to Matches tab. **Application:** when a user complains "I can't find X" or "X should be near Y", treat it as a co-location signal even if the current placement was deliberate at the time it was built. (S048 Issue #13 — Admin Management + Platform Admin moved into AdminDashboard.)

42. **Pure-relocation refactors don't need mockups (CLAUDE.md Workflow Rule #1 has scope).** Mockup-first is for design changes, not container reshuffles. When the JSX is ported verbatim into an existing styled container (no new section header pattern, no new colors, no new typography), there's nothing visually new to review. Confirmed by user opt-in in S048. Don't auto-trigger mockup workflow for component relocations — ask if user wants one when there's any visual ambiguity, otherwise proceed. **Why:** mockup-first protects against unapproved design drift; relocation isn't drift.

43. **`_assert_league_owner(p_league_id)` SECURITY DEFINER helper as first line in every owner-gated RPC.** Single-line `PERFORM public._assert_league_owner(...)` at the top of each function. Cleaner than repeating `IF NOT EXISTS (SELECT 1 FROM leagues WHERE id=... AND created_by=auth.uid()) THEN RAISE EXCEPTION ... END IF;` in every RPC. Centralizes the owner check; future tightening (e.g., 2FA-required) is a one-place change. **Why:** Repeated guard clauses drift; one named function is the right granularity. Same pattern applies to any role tier check used in multiple RPCs. (S050 — applied to 6 new season-management RPCs + update_league_name.)

44. **`position: sticky` + `padding-top: env(safe-area-inset-top)` + missing `html,body{margin:0;padding:0}` reset = "tight when scrolled, gap at top" signature.** When the symptom is "header sits perfectly under the dynamic island when scrolled even a few pixels, but at scrollY=0 there's a visible gap between the safe area and the header", the offset is upstream of the header — a body/html margin or padding inherited from UA defaults or iOS PWA `black-translucent` mode. Sticky's `top:0` masks it once it kicks in but at natural document position (scrollY=0) the offset is exposed. **Fix:** extend any existing `html, body { ... }` CSS injection to include `margin: 0; padding: 0` and add `#root { margin: 0; padding: 0 }` as a defensive baseline. **Why:** Many naive fixes target the header's own padding/positioning and end up over-correcting (S044 spent 3 commits rewriting header padding). The actual offset is on body. Lesson #40 painted body bg + locked overscroll; this lesson completes the pattern with the margin/padding reset. (S049 + S051 together: Lesson #40 hides rubber-band reveals, #44 prevents the natural-position offset itself.) Diagnostic shortcut: if the screenshots show "scrolled state is perfect but scroll-top has a gap", audit body/html FIRST. Don't touch the header.

45. **Wait for the user's screenshot before fixing a multi-attempt visual bug.** When a GitHub issue says "we have attempted to fix this on multiple occasions" or the same surface has been touched repeatedly across recent sessions (S044/S046/S049 all touched the header), pause and use `AskUserQuestion` to capture the actual visible symptom before committing to a fix direction. **Why:** A research report describes the menu of possible fixes; a screenshot tells you which one is actually broken. Without a screenshot, any fix is speculation and risks compounding regressions. (S051 Issue #18: AskUserQuestion offered "wait for screenshot" vs "ship comprehensive defensive fix"; user chose wait, screenshot showed a precise body-margin offset, fix was 1-line CSS instead of a multi-axis defensive sweep.)

46. **Match `<meta name="theme-color">` to the body/header background that owns the top edge.** When `theme-color` differs from the actual top-edge color of the page, iOS Safari/PWA shows a faint seam at launch + during overscroll (visible in the system chrome that briefly tints to `theme-color`). For a sticky header with `linear-gradient(180deg, #0d0d14 0%, ...)` whose top stop is `#0d0d14`, set `theme-color` to `#0d0d14` (NOT the wrapper's general `BG`). Same color discipline as Lesson #40's body bg paint. **Why:** Cosmetic but consistent. The user notices launch-flash even if they can't articulate why it feels off. (S051: index.html `theme-color` #0a0a0f → #0d0d14 alongside the body margin reset.)

47. **For "add X next to every Y" sweeps, anchor the grep on Y (what's already universal), not on existing-X markup.** S052 first push added avatars to 7 PlayerStats slots that already had a letter-circle wrapper — but missed the MOTM Ranking row, which rendered names with no avatar wrapper at all (`{i+1}. {getName(x.pid)}`). The sweep had used the visual "letter circle" pattern as its search target; the missed slot didn't match because it had no circle to find. Required a follow-up commit. **Pattern:** for "add avatar next to every player name" → grep `getName(`, not `width:.*height:.*borderRadius:"50%"`. For "add a tooltip next to every external link" → grep `<a href=`, not existing tooltips. **Why:** existing-X markup is a self-selecting set that excludes the very sites most likely to be missing X. The thing that's already universal (the name, the link, the field name) is the right anchor. (S052: MOTM Ranking avatar followed up in 3b9ef98 after first push e439b0a missed it.)

48. **iOS PWA image upload: `createImageBitmap(file)` is more reliable than `FileReader → data URL → Image()`.** The legacy chain fails intermittently on iOS Safari/PWA, especially with HEIC photos from the Photos library — `img.onerror` fires on first attempt, retry succeeds (browser has cached the conversion). Modern fix: `createImageBitmap` natively decodes any browser-supported format (including HEIC on iOS 17+), produces a drawable usable as the first arg to `ctx.drawImage()`, and skips the FileReader+Image dance entirely. Wrap in a try/catch with `URL.createObjectURL + Image` fallback for older browsers. **Why:** the legacy pattern is from a 2018-era blog post; iOS 17+ has moved on. Same helper used in 2+ upload sites should live in `utils/helpers.js`, not be inlined per-component. (S052 Issue #20 — `decodeImageFile` helper, applied in App.jsx uploadAvatar + EditPlayerModal uploadPhoto.)

49. **Write-through pattern: when user-scoped and entity-scoped state coexist for the same field, update BOTH on write, not at every read.** PadelHub had `profiles.avatar_url` (per-user, set in My Profile) and `players.avatar_url` (per-player, set by admin). Every avatar slot in the app read the player one — so user uploads were invisible. Two strategies: (a) at every read site, look up `profiles.avatar_url` via `player.user_id` if `player.avatar_url` is null; (b) at write time, when the user has a claimed player, update both columns. Picked (b) because it keeps consumers simple (`row.avatar_url`) and the merge happens where context is unambiguous. **Why:** read-time merging spreads the logic across every consumer and is easy to forget to apply at new sites; write-time merging is one well-named function. Trade-off: a tiny bit of stale data if `players.avatar_url` somehow drifts from `profiles.avatar_url` — accepted because the only writers are within the app. (S052: `App.jsx:uploadAvatar` now calls both `profiles.update` and `players.update` then `loadLeagueData`.)

50. **Sidebar/menu entry consolidation: prefer making an existing element clickable over adding a new entry.** Issue #21's framing was "the avatar IS the profile entry" — the most direct interpretation is to wrap the existing avatar + name + email block in a `<button>` and remove the standalone "My Profile" button, not add a new "Go to profile" affordance somewhere. Add a chevron `›` if discoverability is a concern. **Pattern:** when a user complains "X feels like a duplicate" or "X should just live on Y", treat it as a co-location signal (per Lesson #41 / S048). The fix is usually deletion + on-click on the existing surface, not new UI. **Why:** fewer entries, less menu chrome, matches mental model. Same principle as S048 admin entries consolidation. (S052 Issue #21: My Profile sidebar button removed; top user info block now a button with chevron hint.)

51. **When porting a threshold/condition between league-scale and per-player-scale views, recompute whether the threshold semantics still apply at the new scale.** S053 hit this: CombosView's `partnerships.length >= 6` gate (sensible at league scale to avoid noisy "worst" with tiny samples) was copy-pasted into PlayerStats per-player drill-down — where most regular members have <6 distinct partners, hiding worst-pair data even though the league had plenty. **Pattern:** sample-size gates designed for aggregate views are usually too strict for individual views. Default: drop the gate at per-player scope and rely on empty-state JSX to handle the "no data" case. **Why:** the threshold's purpose (statistical confidence in "worst") is league-scale; at per-player scope the user has a specific intuition about their own partnerships and wants to see whatever data exists, however small. (S053 Issue #23 follow-up — `worstPartnership = worstSorted[0] || null` instead of `partnerships.length >= 6 ? worstSorted[0] : null`.)

52. **For destructive UI surfaces, default state should be minimal (a labeled button); reveal full warning copy + confirm flow on tap.** Progressive disclosure beats always-visible safety wall. First version of Danger Zone in S053 shipped with always-visible header + multi-line warning copy + ownership note + delete button — user pushed back as "too much text" because Settings is rarely visited for account deletion specifically. Defensive copy crowds the screen for the 99% of visits that aren't deletions. Better: minimal red-outlined button by default; tap reveals the full red-bordered card with warning + confirm + cancel. **Why:** the safety guarantee comes from the confirm step, not from the always-visible warning. Hide context until intent is signaled. Same pattern as the inline confirm strips used elsewhere in the app. (S053 Issue #22 follow-up — `confirmDelete ? <fullDangerCard/> : <minimalButton/>`.)

53. **Self-delete via Postgres SECURITY DEFINER RPC works when the function owner is `postgres` (default for migrations applied via Supabase MCP/dashboard).** The function runs with the owner's privileges, which include DELETE on `auth.users` — so a single atomic plpgsql function can both clean up public-schema state AND delete the auth row, no Edge Function or service_role client needed. **Pattern shipped:** refuse-if-owner-of-shared-resource (raise EXCEPTION with helpful message); UPDATE NULL on identity columns to preserve history (e.g. `players.user_id = NULL`); DELETE user-scoped state (notifications, push_subscriptions, memberships, profile); DELETE auth.users last. Frontend calls `supabase.rpc("delete_my_account")` then `supabase.auth.signOut()` on success. **Why:** simpler than Edge Function + service_role roundtrip; one place to reason about the deletion; testable as a SQL unit; clear errors propagate to the client via PostgrestException. (S053 Issue #22 — `s053_delete_my_account_rpc` migration.)

44. **Atomic full-replace via DELETE+INSERT for many-to-many edits, with pre-flight validation.** `set_season_roster(p_season_id, p_player_ids[])` validates every passed UUID belongs to the season's league THEN deletes existing rows AND inserts new ones — all in one PL/pgSQL block (implicit transaction). Per-row add/remove RPCs lead to race conditions and partial-failure ambiguity. **Why:** Full-replace is simple, atomic, and matches the user's mental model of "set the roster to this list". Same shape applies to any "edit a many-to-many relationship" UI. (S050.)

45. **Single-active-per-resource enforcement at the RPC layer, not via DB constraint.** A partial unique index `(league_id) WHERE active = TRUE` is the strict version; doing it inside `create_season` / `reactivate_season` lets the RPC also auto-deactivate the previous active season as a side effect. Forgiving flow ("create → previous auto-ends") vs strict gatekeeper ("error: already have active"). **Why:** Match user mental model. Constraint approach forces explicit "end then create" sequence; RPC-level handles it transparently. (S050.)

46. **Self-edit RLS pattern on shared tables: `(membership AND ownership)` USING + `(ownership unchanged)` WITH CHECK.** `players_update_self` lets a claimed user UPDATE their own row only, and the WITH CHECK explicitly blocks `user_id` reassignment (which would let them transfer their claim) and cross-league moves. **Why:** USING gates which rows are visible-for-update; WITH CHECK gates the post-update row state. Both are needed to prevent the "update yourself into someone else's row" attack. Pair with `players_claim_self` (initial claim when `user_id IS NULL`) and `players_update_admin` (admin scope). (S050.)

54. **After context compaction, verify that file edits described in the summary were actually written to disk before building on top of them.** S054 resumed from a compacted session whose summary described "#24 + #26 changes applied to App.jsx, theme.js, Sidebar.jsx" — but none of those files were actually modified. The session had run out of context before the Edit calls fired. **Pattern:** on resumption from compaction, grep one key string from the "applied" summary (e.g., `grep "repeat(5" App.jsx`) before assuming the disk state matches the summary's claims. If it's missing, apply the changes fresh. **Why:** session summaries describe intent, not confirmed execution. The compaction boundary truncates anything after the last summary-point — an Edit call that fired 3 seconds before compaction might not make it into the summary. (S054: App.jsx still had `repeat(7,1fr)` and original LazyFallback after summary claimed both were changed.)

57. **Never use a separator that can appear inside the values being joined.** UUIDs contain hyphens (`-`), so using `join('-')` to build a composite key from two UUID IDs produces an ambiguous string — `split('-')` then returns many fragments instead of two IDs, causing lookups to return `undefined`/"?". **Rule:** use a character guaranteed absent from the value domain as the separator. For UUID keys: `'|'`, `'~'`, or `'#'` are safe. For arbitrary strings: `'\0'` (null byte). Pattern: `[id1, id2].sort().join('|')` / `key.split('|')`. (S056: Season Awards Top Pair showed "??" for both player names because UUIDs split on '-' yielded 10 fragments.)

56. **Push to git directly without asking the user for permission.** User explicitly instructed: "never ask me to push myself. just push it directly yourself." This is a standing rule — never stop to ask "should I push?" or "do you want me to push?". After verified changes are ready: commit, push, report the result. **Why:** Interrupting for push confirmation adds friction with zero benefit; the user is already trusting Claude to make the code changes. If a destructive action (force-push, reset) is involved, that's a separate concern — but normal `git push origin main` should be done autonomously. (S055 — user feedback mid-session.)

55. **`Promise.allSettled` over `Promise.all` when parallel API calls can partially fail.** `Promise.all` abandons all results if any one rejects — e.g., `platform_get_users` timing out means stats and leagues also fail. `Promise.allSettled` resolves after all settle, with `{status:"fulfilled",value}` or `{status:"rejected",reason}` per call. Frontend can then set whichever partial data arrived. Pair with a loadError state that activates only when ALL calls fail (not just one), and a Retry button on the error state. **Why:** Platform Admin is a diagnostics tool — showing partial data is far more useful than a blank screen. The RPC most likely to time out (`platform_get_users` — queries auth.users) shouldn't block stats from rendering. (S054 Issue #29 — PlatformAdmin.jsx `loadData` rewrite.)

47. **Searchable combobox > native `<select>` above ~30 entries.** Native first-letter-jump on iOS Safari is unreliable and non-discoverable on desktop. A 100–200-line custom component (text input + filtered scrollable list + click-outside-to-close + auto-focus on open) is the right investment when the list grows past ~30. Pattern from `CountrySelect.jsx` (S050): controlled `value`/`onChange` props, internal `query` state for filter, `useRef` + `useEffect` mousedown/touchstart listener for outside-click close, a "— Not set —" / clear option always at the top. **Why:** Scrolling 195 options to find Palestine vs typing "pal" is a UX gulf, not a polish gap. Reusable for any large enum picker. (S050.)

48. **Per-word startsWith filter for multi-word names.** `name.split(/[\s-]+/).some(w => w.startsWith(q))` finds "South Korea" when typing "korea" and "Czech Republic" when typing "republic". Plain `name.startsWith(q)` would miss both. Plain `name.includes(q)` produces irrelevant matches (typing "p" pulls "Cabo Verde" / "Egypt" / "Japan"). **Why:** Multi-word names have meaningful tokens beyond the first word; users search by any of them. Hyphens treated as word boundaries via `/[\s-]+/`. (S050 CountrySelect filter.)

58. **Mockup-first iteration is the right cadence for icon design — do NOT implement first.** S057 went through 3 mockup rounds for Issue #36 nav icons (v1 generic outline icons → v2 added 3-silhouette Players + padel teardrop racket → v3 emoji-style trophy + full-tab oval capsule active state). Each round took 1 user feedback turn at zero implementation cost. Implementing in code first would have meant editing source files, restarting dev server, screenshotting, then iterating in the actual app — far more friction per round. **Pattern:** for any visual-design task touching multiple SVGs / colors / layout, build an HTML mockup in `padelhub/public/mockup-issue<N>.html` showing 2-4 variants side-by-side with annotated trade-offs; iterate via Edit on the mockup until approved; only THEN port to source. (S057 Issue #36.)

59. **Sync derived selection state via shared Context, not duplicate per-component state.** S057 Issue #40 — when the same conceptual filter (current season) drives multiple screens, lifting it into the LeagueContext provider eliminates "why did my selection reset?" surprises. Trade-off accepted: any consumer can mutate the shared state — fine when there's a small, well-defined set of mutators (in this case, two `<select>` dropdowns on different screens both calling `setSelectedSeason`). Alternative considered: independent state per screen — rejected because user explicitly noted "switching season on Ranking didn't propagate to Matches" as the bug. (S057.)

60. **Single-dispatcher icon component over per-icon imports in nav consumers.** `<NavIcon name="trophy" active={…}/>` lets App.jsx import one symbol; the dispatcher routes by string. Adding a 5th icon requires editing only NavIcons.jsx + the theme.js icon string — App.jsx is untouched. Trade-off vs. per-icon imports: a tiny bit of indirection, but pays off when icons are referenced via data-driven config (theme.js TL/TR arrays). Same pattern as a sprite-by-key icon set. (S057.)

61. **Before bumping any version-numbered constant in a multi-PC repo, run `diff` against git first OR pull latest before reading.** S057 hit a near-miss: local `sw.js` showed `v71` but git was already at `v73` (S056 had bumped during my session via the parallel "user closed #37/#38/#39" thread). ## Successful Patterns (carried forward)

- **Plain object Context** — safer than useMemo for values with derived state. Pattern: `const ctx = { ...values };` not `useMemo(() => ({...}), [deps])`.
- **SECURITY DEFINER + FOR UPDATE for atomic multi-user state** — eliminates read-modify-write races. Client sends intent, server handles mutation atomically with row lock.
- **Debounced realtime via `useRef` timer + `useCallback`** — no lodash dep, auto-cleanup on unmount. `const ref=useRef(null); const fn=useCallback(()=>{clearTimeout(ref.current);ref.current=setTimeout(action,500);},[]);`
- **Tiebreaker sort composition via `||`** — `(a,b) => primaryCmp(a,b) || secondaryCmp(a,b)`. Second runs only when first returns 0. Chains cleanly.
- **Independent ASC sort for "worst" extreme** — never `sortedDesc[length-1]`. Pair with `worst.value < best.value` gate to hide when everyone ties.
- **Component extraction with view-local state** — moving useState + functions exclusive to one view → self-contained component, zero cross-view break risk. Identify state never read by other views.
- **Reusable component with callback handoff over UI duplication** — `<TeamShuffler onAccept={({matches, sitouts}) => ...} />` lets each parent decide what to do with the shuffle result. Single UI, multi-consumer.
- **Inline HTML splash screen** in `index.html` — pure HTML/CSS, zero JS dep. Renders even if CDN is down. Prevents the "blank root div" problem.
- **`Promise.race` for auth timeout** — cleaner than `setTimeout`+flag. Pattern: `Promise.race([actual(), new Promise((_,r)=>setTimeout(()=>r('timeout'),5000))])`.
- **`postMessage` from SW to clients for update notification** — SW `activate` → `clients.matchAll()` → `postMessage({type:'SW_UPDATED'})` → client `location.reload()`. No external libs.
- **`isRecoveryRef` (useRef) for cross-render state in long-lived listeners** — useState captured in useEffect closures goes stale; useRef.current is always fresh.

Trigger-level `RAISE EXCEPTION` on the strict path catches new client-driven violators that try to claim incomplete status without legitimate reason. **Why:** Tightening a CHECK on a populated table is otherwise destructive (existing rows must conform); conditional bypass + trigger-level enforcement is the gentler defence-in-depth pattern.
- **IMMUTABLE wrapper function for CHECK constraints needing subquery logic (S045):** `CREATE FUNCTION shape_valid(jsonb) RETURNS boolean LANGUAGE sql IMMUTABLE SET search_path = public AS $$ SELECT bool_and(...) FROM jsonb_array_elements(...) ... $$;` then `CHECK (shape_valid(col))`. Postgres CHECK constraints forbid subqueries directly — wrapping the validation in an IMMUTABLE SQL function is the clean workaround. **Why:** Lets composite shape rules express in the constraint without resorting to triggers, and the function is reusable in audit queries / RPCs.

Maps cleanly to UX semantics: admin+complete=approved, admin+incomplete=incomplete, non-admin+complete=pending, non-admin+incomplete=incomplete. **Why:** Single point of truth for what status a new row gets. Migration becomes idempotent — re-running has no effect after first application. **Why:** Self-documenting (reviewer sees what was rectified to what) and safe to re-run if migration log uncertainty arises.

88. **Late useCallback declaration → TDZ on useEffect deps.** When adding a `const fn = useCallback(...)` mid-component, ensure it's declared BEFORE any earlier `useEffect(..., [fn, ...])` that references it in its dep array. The helper:

  function editFile(p, edits) {

    for (const [o, n, label] of edits) {
      if (!s.includes(o)) { fails.push(label); continue; }
      s = s.replace(o, n);

    if (had) s = s.replace(/\r?\n/g, '\r\n');
    fs.writeFileSync(p, s);

## Lessons Learned (S077 mega-session additions)

- [2026-05-11] **Lesson #106 — `node -e "<heredoc>"` is unsafe for files with `${}` / backticks.** Multi-line edit scripts invoked via `node -e "..."` from bash get the heredoc content shell-interpolated *before* node receives it. Variables like `${activeSeason.name}` or template-literal backticks inside the file content get partially expanded by bash, the script appears to run cleanly (file write returns ok) but the pattern didn't actually match what was on disk. **Prevention rule: ALWAYS write edit scripts to a `/tmp/*.js` file via the Write tool first, then run `node /tmp/edit.js`.** The Write tool delivers the JS source verbatim — no shell interpolation, no quoting bugs. - [2026-05-11] **Lesson #107 — Retry-with-backoff masks slowness instead of fixing it.** Wrapping a slow operation in `try first; on error, wait 350ms; retry` is a comfort blanket if the first attempt is genuinely transient, but if the first attempt is *hanging* on a slow underlying operation (RLS cascade, network timeout, blocked trigger), each retry adds the full request-timeout to the perceived latency. **Prevention rule: retries-with-backoff MUST set an explicit per-attempt timeout AND surface progress (loading state, progress text). If the user reports "this is slow", suspect the retry layer FIRST. When in doubt, drop the retry and let the error surface — the fix is at the root.** (S077 r15 — added retry on delete_season expecting transient errors; user reported "30 seconds to delete". - [2026-06-19] **"new_profile" join type ≠ "new to the app" — label by whether the user already plays elsewhere.** Existing PadelHub members joining a second league still use `type='new_profile'` (a new player row is minted), so the queue mislabeled them "NEW PROFILE". - [2026-06-19] **A user's player row is created on EVERY path into a league — enumerate them all when adding profile carry-over.** #108 fixed the JOIN paths (`approve_join_request`), but the user then found their avatar missing in a league they OWNED: `create_league` auto-creates the owner's player with only name+user_id. Both the join AND the create paths needed the same backfill (migration `s108b`). **Why:** "carry the user's data into a new league" isn't one place — it's wherever a `players` row is first minted for that user (approve_join_request for members, create_league for owners). Per-league avatars (the app stores avatar on each player row, S051) make this a recurring trap. Note the global `profiles.avatar_url` (header/Sidebar/ProfileView) was fine; only the per-league player rows drifted.

- [2026-06-20] **Hex→token color sweep: exclude the `:root` definition line or the token resolves to itself.** A blind `replace_all "#9090a4" → "var(--muted)"` rewrites the definition `--muted: #9090a4;` into `--muted: var(--muted);` (circular → unresolved → every consumer breaks). Skip the line matching `/^\s*--<token>:\s*#hex/` and verify post-sweep with `grep` expecting exactly 1 remaining hit (the definition). **Why:** the failure is silent at edit time and only shows as missing colors at runtime. Also (Lesson #69): only sweep byte-equal pairs — `#5a5a6a` has no equal token, so mapping it is a visual change, not a refactor; leave it.
- [2026-06-20] **For "make X global across leagues", propagate via a SECURITY DEFINER RPC, not client UPDATEs.** #110 (global profile): an admin editing a player can't reach that user's rows in OTHER leagues under RLS, so the propagation must run with definer privileges. One `sync_player_identity(p_player_id)` RPC authorizes (linked user OR source-league admin), copies identity fields to all the user's claimed rows, and centralizes the one exception (self-grade propagates; admin grade override stays local). Called after the save at all three edit sites. **Why:** identity is per-user but stored per-league (`players`), so "global" = write-through to siblings; the RPC is the single authorized place, and new-league joins already carry over (S087) so the loop closes without a backfill. Same family as Lesson #49 (write-through over read-time merge).
- [2026-06-20] **A propagation/fan-out UPDATE must COALESCE, never flat-assign — or a sparse source row erases good data in the targets.** #110's first `sync_player_identity` did `SET avatar_url = v_src.avatar_url, …` from the just-edited row. But per-league `players` rows drift (avatar write-through only touches the current league — S087), so a grade self-assessment saved in a league whose `avatar_url` is NULL would propagate that NULL and **wipe the avatar in every other league**. Fix: `SET avatar_url = COALESCE(v_src.avatar_url, p.avatar_url)` for every identity field (migration `s109b`) — a real value wins, NULL never clears. **Why:** the triggering edit rarely touches every field; flat-assigning all of them treats "unset here" as "clear everywhere". Caught in review before user impact; esbuild/grep/deploy-READY can't catch a data-logic bug — trace the RPC with a NULL source field against a populated sibling. Same COALESCE discipline as `approve_join_request` (S087).
- [2026-06-20] **A "two loading screens flash" on app launch = a static (index.html) splash and a framework (React) splash that differ.** PadelHub's flashed because the two splashes had different bg (#080808 vs #0d0d14), logo size (160 vs 140), and font (index.html DECLARED `font-family:'Syne'` but never LOADED Syne → system fallback, while React had real Syne). Fix = make them byte-identical: same bg, same logo size, same font (and actually load it in index.html), and give the framework splash the same wordmark. **Why:** the flash IS the visible handoff between two near-identical-but-not screens; eliminate the diff, eliminate the flash. (S088 logo sweep.)
- [2026-06-20] **A brand mark is duplicated across React components + standalone SVG files + raster PNGs — map EVERY surface before a logo swap, or stale copies survive.** PadelHub's `favicon.svg` and `og-image.png` (the WhatsApp/iMessage link preview) were still the OLD grey racket long after the orb shipped everywhere else, precisely because nothing in-app renders them so they never visibly drifted. Build the full surface map first (`planning/logo-sweep.md`): React mark components, index.html inline splash, `icon.svg`/`favicon.svg`, the PNGs, manifest/sw refs, og cache-buster + alt text. Keep the canonical mark in ONE component (designer swap later = one file). Rasterize PNGs from the SVG with `sharp` installed in a scratch dir (repo deps untouched), and VIEW the generated PNGs (Read tool renders them) before shipping. (S088.)

## Patterns That Work (S089 additions)
- [2026-06-20] **Live-preview from the working clone via a dedicated `clone-dev` launch.json** (node `-e` chdir to `C:\Users\User\dev\Padel-Battle`, run vite on port 5182) — Why: the OneDrive mirror has no `node_modules`; this is the only reliable way to visually verify the preview against CURRENT code. Reuse it every session.
- [2026-06-20] **Self-contained `ConfirmButton` wrapper** (owns its own modal open-state) — Why: let 4 tournament files swap `<button onClick={()=>{if(confirm())fn()}}>` → `<ConfirmButton onConfirm={fn}>` with ZERO per-file state plumbing. Good pattern for replacing native `confirm()` app-wide.
- [2026-06-20] **Bake icon transforms into Icon.jsx, not per-call-site** — Why: e.g. the handedness 45° tilt added once to `hand-left`/`hand-right` cases applies on every screen automatically (drill-in, profile, admin, onboarding) — guarantees consistency, no missed call sites.
- [2026-06-20] **Add a `size` prop to a shared control instead of CSS-overriding it** — ScoreStepper's buttons are inline-styled 40px (CSS can't override inline); a `size` prop made compact 28px steppers possible inside the tight round card. When a shared component is too big for a new context, parametrize it.
- [2026-06-20] **Build approved-mockup features, then verify against a LIVE instance** — the round flashcard looked fine in the static mockup but the live steppers crowded names; only a live 4-player Americano revealed it. Mockups approve layout; live verification catches interaction/space realities.
